Arquitetura do Janus

A arquitetura do Janus é projetada com uma filosofia modular e desacoplada, inspirada em microsserviços e sistemas de agentes inteligentes. Isso garante escalabilidade, resiliência e evolução independente dos componentes. A estrutura lógica da aplicação (app/) é dividida em camadas claras: Interface (API), Lógica de Negócios (Serviços), Núcleo Inteligente (Core), Acesso a Dados (Repositórios) e Persistência (Banco de Dados).

Arquitetura de Comunicação (Chat)

O chat textual opera exclusivamente via API REST (`/api/v1/chat/*`), garantindo compatibilidade e previsibilidade. A camada WebRTC é usada opcionalmente para mídia (áudio/vídeo) no navegador, inicializada com Janus (`Janus.init`, `Janus.attach`) e configurada com `RTCPeerConnection`. Em indisponibilidade de WebRTC, o sistema mantém o chat REST integralmente.

  • Primário: REST para mensagens de chat e histórico.
  • Opcional: WebRTC para mídia; sem uso de datachannel para texto.
  • Fallback automático para REST quando houver falhas de conexão WebRTC.

Componentes Principais por Camada

Explore as camadas da aplicação app/ por meio do acordeão abaixo.

Fundação: Configuração e Ponto de Entrada

O arquivo main.py inicializa a aplicação FastAPI, carrega as configurações de config.py e monta os roteadores. Esta fundação provê inicialização previsível e configurações centralizadas.

Camada de API: A Porta de Entrada

Responsável por requisições HTTP, validação e autenticação. Define os schemas de entrada e saída e expõe os endpoints públicos.

  • Endpoints principais: endpoints/chat.py, endpoints/agent.py, endpoints/knowledge.py, endpoints/autonomy.py, etc.
Camada de Serviços: Orquestração

Orquestra tarefas, consome o Core e os Repositórios, aplica regras de negócio e consolida respostas.

  • Serviços-chave: ChatService, MetaAgentService, KnowledgeService, OptimizationService.
O Núcleo (Core): O Cérebro da IA

A lógica fundamental da IA. Componentes principais:

  • core/agents: MetaAgent e MultiAgentSystem.
  • core/autonomy: GoalManager, Planner, PolicyEngine.
  • core/memory: KnowledgeGraphManager, GraphRAGCore, WorkingMemory.
  • core/workers: Processamento assíncrono (Orchestrator e workers especializados).
  • core/optimization: ReflexionCore, SelfOptimization.
  • core/monitoring: HealthMonitor, AutoHealer.
  • core/infrastructure: PythonSandbox, MessageBroker.
Camada de Repositórios: Abstração de Dados

Aplica o Repository Pattern para desacoplar a lógica de negócios da fonte de dados, fornecendo contratos estáveis e testáveis.

Camada de Persistência: A Memória Física

Tecnologias de banco de dados utilizadas:

  • vector_store.py: Banco Vetorial (ex.: Qdrant).
  • graph.py: Banco de Grafo.
  • mysql_config.py: Banco Relacional/SQL.
Modelos e Esquemas: A Linguagem Comum

O módulo app/models define modelos e schemas (schemas.py, Pydantic) que garantem contratos de dados consistentes entre API, Serviços, Core e Repositórios.
