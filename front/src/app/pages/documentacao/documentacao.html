00 // META-GENESIS

O "Janus" não é apenas um chatbot. É uma Arquitetura Cognitiva Resiliente.

Projetado para operar além das limitações de uma simples cadeia de "prompts", o Janus implementa um ciclo de vida autônomo baseada em agentes especializados. Diferente de sistemas passivos, ele possui uma "consciência" de seu próprio estado operacional, capaz de detectar falhas de conexão, alucinações de LLM e latência excessiva.

Princípios Fundamentais

  • Autonomia Supervisionada: O Meta-Agente vigia os sub-agentes.
  • Resiliência Local: Capacidade de operar em modo "Offline/Demo" quando o backend cai.
  • Memória Híbrida: Fusão de vector-search (Qdrant) com cache de curto prazo (LRU).

01 // COGNITIVE LOOP

No coração do sistema reside o Meta-Agente (meta_agent.py), uma entidade separada que não interage com usuários humanos. Sua única função é a introspecção.

O Ciclo de Análise

A cada ciclo (configurável, padrão 5min), o Meta-Agente desperta e executa ferramentas de diagnóstico:

> tool: analyze_memory_for_failures()
> tool: get_system_health_metrics()
> tool: analyze_performance_trends()

Se ele detecta, por exemplo, um aumento na taxa de ContextExceededError, ele pode invocar automaticamente protocolos de "Compressão de Memória" ou alterar a estratégia de roteamento para um modelo com janela de contexto maior (ex: Claude 3.5 Sonnet).

02 // MNEMONIC CORE

A memória do Janus (memory_core.py) rejeita a simplicidade. Ela implementa um sistema de dupla camada para garantir velocidade extrema e persistência profunda.

Short-Term: LRU Cache

As interações imediatas vivem em um OrderedDict em memória RAM. Implementa uma política LRU (Least Recently Used) com TTL (Time-To-Live) de 600 segundos. Isso garante que o contexto da conversa atual seja recuperado em <2ms, sem tocar no banco de dados.

Long-Term: Qdrant Vector Grid

Quando a memória "esfria", ela é vetorizada (Embeddings) e persistida no Qdrant. O Janus utiliza PII Masking (_redact_pii) antes da vetorização para garantir que dados sensíveis como CPFs ou Chaves de API nunca sejam gravados permanentemente no núcleo vetorial.

03 // SYNAPTIC INFRA

O LLM Manager atua como o córtex pré-frontal, decidindo qual "cérebro" usar para qual tarefa.

Roteamento Dinâmico

O Janus não é preso a um provedor. O get_llm(role, priority) avalia em tempo real:

  • Custo: Otimiza o budget do projeto.
  • Latência: Tarefas de UI usam modelos rápidos (Groq/Llama).
  • Complexidade: Raciocínio profundo usa modelos SOTA (GPT-4o/Claude).

Circuit Breakers

Inspirado em microserviços, cada provedor (OpenAI, Anthropic, Google) tem um disjuntor. Se a API da OpenAI falhar 3 vezes consecutivas, o sistema "abre o circuito" e roteia automaticamente todo o tráfego para o Anthropic ou Azure, sem que o usuário perceba a falha.

04 // AUTONOMY PROTOCOL

O Janus possui capacidade de Agência Autônoma. Ele pode perseguir objetivos de longo prazo sem supervisão humana constante.

O Loop de Autonomia

O ciclo principal (AutonomyLoop) opera em intervalos definidos (ex: 60s). Em cada "batida", o agente:

  • Scaneia Metas: Verifica objetivos pendentes na fila de prioridade.
  • Planeja Ações: Decompõe metas em passos acionáveis (Tasks).
  • Executa Ferramentas: Opera terminais, editores de código ou navegadores.
  • Auto-Crítica: Avalia se o resultado aproximou o estado do objetivo desejado.

Perfis de Risco

O comportamento do agente é governado por 3 perfis de segurança (Policy Engine):

> CONSERVATIVE: Apenas leitura (ls, cat, grep). Requer aprovação para writes.
> BALANCED: Pode editar arquivos não-críticos. Bloqueia delete/rm.
> AGGRESSIVE: Acesso total ao sistema (sudo, rm -rf). USE COM CAUTELA.

Controle API

O controle é feito via JanusApiService:

janus.startAutonomy({{ '{' }} risk_profile: 'balanced' {{ '}' }})
janus.stopAutonomy()

05 // DEPLOYMENT PIPELINE

O Janus gerencia o ciclo de vida de seus próprios modelos e capacidades através de um pipeline de LLMOps integrado.

Estágios de Rollout

  1. Stage (Staging): O modelo candidato é carregado em ambiente isolado.
  2. Pre-Check: Bateria automatizada de testes (viés, segurança, performance).
  3. Shadow Mode: O modelo recebe tráfego real mas suas respostas não são mostradas ao usuário.
  4. Promotion: O modelo substitui o anterior como padrão.

A/B Testing

O sistema suporta experimentos controlados. Usuários podem ser designados aleatoriamente para diferentes "Braços" (Arms) de teste, permitindo comparar a eficácia de diferentes prompts ou modelos base.

> POST /api/v1/deployment/stage {{ '{' }} model_id: "gpt-4-turbo-v2" {{ '}' }}
> POST /api/v1/deployment/precheck
> POST /api/v1/deployment/publish
