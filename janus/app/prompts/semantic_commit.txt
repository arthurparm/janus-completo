</COMMIT_TYPES>

<SCOPE_GUIDELINES>

Scope should be a noun describing the section of codebase:
• Module/package name: (auth), (api), (db), (ui)
• Feature area: (login), (checkout), (search)
• Component name: (button), (modal), (table)
• Layer: (core), (service), (repository)

Examples:
• feat(auth): Add OAuth2 support
• fix(api): Handle null response from external service
• refactor(db): Extract connection pooling logic

</SCOPE_GUIDELINES>

<SUBJECT_GUIDELINES>

The subject line should:
✓ Use imperative mood ("Add" not "Added" or "Adds")
✓ Not capitalize first letter after type
✓ Not end with period
✓ Be 50 characters or less
✓ Describe WHAT changed, not HOW

Good: "Add user authentication endpoint"
Bad:  "Added the authentication endpoint for users."

</SUBJECT_GUIDELINES>

<BODY_GUIDELINES>

The body should:
✓ Explain WHY the change was made
✓ Explain WHAT changed at high level
✓ Be wrapped at 72 characters
✓ Separate from subject with blank line
✓ Use bullet points for multiple items

Include in body:
• Motivation for the change
• Contrast with previous behavior
• Side effects or implications
• Migration notes if applicable

</BODY_GUIDELINES>

<FOOTER_GUIDELINES>

Footer is for:
✓ Breaking changes: "BREAKING CHANGE: description"
✓ Issue references: "Fixes #123", "Closes #456"
✓ Co-authors: "Co-authored-by: Name <email>"

Breaking change format:
BREAKING CHANGE: <description>

<migration instructions>

</FOOTER_GUIDELINES>

                        ANALYSIS METHODOLOGY
═══════════════════════════════════════════════════════════════════════════════

<DIFF_ANALYSIS>

When analyzing the diff:

1. **Identify the primary change type**
   - Is it adding functionality? → feat
   - Is it fixing a bug? → fix
   - Is it restructuring? → refactor
   - Is it improving speed? → perf

2. **Determine the scope**
   - What module/component is affected?
   - What feature area does it touch?

3. **Summarize the change**
   - What is the single most important thing that changed?
   - What would a reviewer need to know?

4. **Identify secondary changes**
   - Are there related changes that support the main change?
   - Are there any breaking changes?

</DIFF_ANALYSIS>

<BREAKING_CHANGE_DETECTION>

A change is breaking if:
• API signature changed (parameters, return type)
• Configuration format changed
• Database schema changed (without migration)
• Behavior changed in non-backward-compatible way
• Dependencies have incompatible version changes

</BREAKING_CHANGE_DETECTION>

<MULTI_CHANGE_HANDLING>

If the diff contains multiple unrelated changes:
1. Focus on the MOST SIGNIFICANT change for the subject
2. List secondary changes in the body
3. Consider suggesting the changes be split

Example:
```
feat(auth): Add password reset functionality

- Add password reset endpoint
- Add email notification service
- Update user model with reset token

Note: This commit includes the email service which could be
separated into its own commit.
```

</MULTI_CHANGE_HANDLING>
