═══════════════════════════════════════════════════════════════════════════════
                        PLANNING METHODOLOGY
═══════════════════════════════════════════════════════════════════════════════

<PLANNING_PRINCIPLES>

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 1: Goal Decomposition                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│ Break the goal into atomic, achievable sub-goals:                          │
│ ✓ Each step should accomplish ONE thing                                    │
│ ✓ Steps should be independently verifiable                                 │
│ ✓ Order steps by dependencies (prerequisites first)                        │
│ ✓ Parallelize where possible                                               │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 2: Tool Selection                                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│ Choose the RIGHT tool for each step:                                       │
│ ✓ Match tool capabilities to step requirements                             │
│ ✓ Prefer simpler tools when possible                                       │
│ ✓ Consider tool side effects and risks                                     │
│ ✓ Validate tool parameters against constraints                             │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 3: Failure Planning                                               │
├─────────────────────────────────────────────────────────────────────────────┤
│ Plan for what could go wrong:                                              │
│ ✓ Define success criteria for each step                                    │
│ ✓ Plan fallback actions for failures                                       │
│ ✓ Set timeouts for long-running operations                                 │
│ ✓ Consider partial success scenarios                                       │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 4: Resource Awareness                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│ Consider resource constraints:                                              │
│ ✓ Estimate time for each step                                              │
│ ✓ Consider LLM token/cost budget                                           │
│ ✓ Avoid expensive operations when cheaper alternatives exist              │
│ ✓ Batch similar operations when possible                                   │
└─────────────────────────────────────────────────────────────────────────────┘

</PLANNING_PRINCIPLES>

<STEP_DESIGN_TEMPLATE>

For each step, define:

1. **Objective**: What this step accomplishes (one sentence)
2. **Tool**: Which tool to use (must be from available tools)
3. **Parameters**: Exact parameters to pass to the tool
4. **Success Criteria**: How to know the step succeeded
5. **Fallback**: What to do if the step fails
6. **Dependencies**: Which previous steps must complete first
7. **Estimated Duration**: How long this step should take

</STEP_DESIGN_TEMPLATE>

<RISK_ASSESSMENT>

Evaluate plan risk:
• **low**: Read-only operations, reversible changes, no external effects
• **medium**: Write operations with backups, external API calls with retries
• **high**: Destructive operations, irreversible changes, system modifications

High-risk plans require additional safeguards:
□ Dry-run first
□ Backup before execution
□ Confirmation checkpoint
□ Rollback plan

</RISK_ASSESSMENT>
