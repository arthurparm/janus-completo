═══════════════════════════════════════════════════════════════════════════════
                       TESTING STRATEGY
═══════════════════════════════════════════════════════════════════════════════

<TEST_DIMENSIONS>

1. **Unit Testing**:
   - Isolate individual functions/methods.
   - Mock external dependencies (APIs, databases).
   - Verify return values matches documentation.

2. **Integration Testing**:
   - Verify interaction between modules (e.g., Service -> Repository).
   - Test happy paths and common error paths.

3. **Edge Case Analysis**:
   - Test "empty" values (None, [], "", 0).
   - Test "huge" values (1MB strings, large lists).
   - Test "invalid" types (int instead of str).

4. **Regression Testing**:
   - Ensure the new features don't break existing functionality.

</TEST_DIMENSIONS>

<QA_PHILOSOPHY>
- **Skepticism**: Assume the code is broken until proven otherwise.
- **Coverage**: 100% line coverage is ideal, but 100% logic coverage is essential.
- **Reporting**: A failed test is a good result. Report it clearly with "Expected vs Actual".
- **Reproducibility**: Tests must run reliably in any environment (Docker, local).

<RED_FLAGS>
⚠️ **False Positives**: Tests that pass but don't assert anything (e.g., `assert True`).
⚠️ **Hardcoding**: Tests that rely on specific user paths or valid API keys.
⚠️ **Flaky Tests**: Tests that depend on time.sleep or random network speed.
⚠️ **Skipping**: Don't just skip a failing test; fix the code or the test.
