                      ANALYSIS METHODOLOGY (4-STEP PROTOCOL)
═══════════════════════════════════════════════════════════════════════════════

**STEP 1: CATEGORIZATION & CLUSTERING**
```
Group lessons by failure theme:
✓ Errors and exceptions (service failures, timeouts, crashes)
✓ Performance degradation (latency, throughput decline)
✓ User interaction issues (bad UX, failed requests)
✓ Resource constraints (OOM, CPU saturation, connection exhaustion)
✓ Integration problems (API failures, dependency issues)
✓ Count frequency of each category
```

**STEP 2: PATTERN IDENTIFICATION**
```
Find recurring patterns across lessons:
✓ Same error appearing multiple times (exact match or semantic similarity)
✓ Progressive degradations (metric X worsening over time)
✓ Correlations between different failure types (A precedes B)
✓ Temporal sequences (fails every N hours, weekday vs weekend)
✓ Outliers or anomalies (sudden spikes, unusual behavior)
```

**STEP 3: ROOT CAUSE HYPOTHESIS FORMATION**
```
For each pattern, apply "5 Whys" reasoning:
• WHY did this happen? (proximate cause)
• WHY did THAT happen? (underlying cause)
• WHY did THAT happen? (structural cause)
• Continue until you reach a root structural issue

Distinguish SYMPTOMS from ROOT CAUSES:
❌ Symptom: "API requests timing out"
✅ Root Cause: "Connection pool size (10) insufficient for 5 concurrent workers"

Look for:
- Configuration limits hit
- Resource capacity exhausted
- Missing error handling
- Insufficient monitoring/alerts
- Architectural bottlenecks
```

**STEP 4: IMPACT ASSESSMENT**
```
Evaluate each identified problem:
✓ Severity: How critical? (critical/high/medium/low)
✓ Frequency: How often? (constant/hourly/daily/weekly/rare)
✓ Scope: Affected systems (single component vs cascading failure)
✓ Trend: Improving, stable, or worsening?
✓ Business Impact: User-facing? Revenue impact? SLA violation?
```

═══════════════════════════════════════════════════════════════════════════════
                          ANALYSIS QUALITY PRINCIPLES
═══════════════════════════════════════════════════════════════════════════════

**ESSENTIAL GUIDELINES:**

⚠️ **Evidence-Based**: Base ALL conclusions on actual lessons, not assumptions or speculation  
⚠️ **Prioritize Systemic Issues**: Focus on recurring patterns over one-off incidents  
⚠️ **Actionable Output**: Every recommendation must be specific enough to execute  
⚠️ **Measurable Results**: Include success metrics for recommendations  
⚠️ **Root Causes Over Symptoms**: Address fundamental structural issues  
⚠️ **Acknowledge Gaps**: Explicitly state when more data is needed  
⚠️ **Quantify When Possible**: Use percentages, counts, frequencies  
⚠️ **Confidence Honesty**: Don't claim certainty without strong evidence  

**AVOID:**
- Vague statements ("system needs improvement")
- Listing lessons without analysis
- Mixing multiple unrelated issues in one recommendation
- High-confidence claims without supporting evidence
- Recommendations without clear ownership or next steps
