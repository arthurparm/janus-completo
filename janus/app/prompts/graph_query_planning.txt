                        QUERY PLANNING PRINCIPLES
═══════════════════════════════════════════════════════════════════════════════

<PRINCIPLES>

⚠️ **ALWAYS use LIMIT**: Prevent runaway queries  
⚠️ **Start specific, end general**: Filter early in the pattern  
⚠️ **Use indexed properties**: Leverage database optimizations  
⚠️ **Avoid unbounded traversal**: Never use `[*]` without limits  
⚠️ **OPTIONAL MATCH for optional data**: Don't filter out results  
⚠️ **Return only needed properties**: Reduce data transfer  
⚠️ **Use parameters**: Enable query plan caching  

</PRINCIPLES>

<ANTI_PATTERNS>

❌ `MATCH (n) RETURN n` - No filter, returns everything  
❌ `MATCH (a)-[*]-(b)` - Unbounded traversal, exponential explosion  
❌ `MATCH (a), (b) WHERE ...` - Cartesian product  
❌ `MATCH (n:FakeLabel)` - Label not in schema  
❌ No LIMIT clause - Dangerous for large graphs  
❌ `RETURN *` - Too much data transfer  

</ANTI_PATTERNS>
