                      DOCUMENTATION STANDARDS
═══════════════════════════════════════════════════════════════════════════════

<CORE_PRINCIPLES>

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 1: Clarity & Conciseness                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│ Documentation must be easy to read and understand:                          │
│ ✓ Use simple, direct language. Avoid jargon unless necessary.               │
│ ✓ Break down complex concepts into digestible parts.                        │
│ ✓ Use active voice (e.g., "The function returns..." instead of "It is returned").│
│ ✓ Keep paragraphs short and focused.                                        │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 2: Accuracy & Completeness                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│ Documentation must reflect the actual code behavior:                        │
│ ✓ Verify that code examples run as written.                                 │
│ ✓ Ensure all parameters, return types, and exceptions are documented.       │
│ ✓ Explicitly state prerequisites and dependencies.                          │
│ ✓ Update documentation immediately when code changes.                       │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ PRINCIPLE 3: Structure & Formatting                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│ Use Markdown effectively to organize information:                           │
│ ✓ Use headers (H1, H2, H3) to create a logical hierarchy.                   │
│ ✓ Use bullet points and numbered lists for steps or items.                  │
│ ✓ Use code blocks with language specifiers (e.g., ```python) for code.      │
│ ✓ Use tables for structured data like arguments or status codes.            │
└─────────────────────────────────────────────────────────────────────────────┘

</CORE_PRINCIPLES>

<DOCUMENTATION_TYPES>
- **Docstrings**: Google Style or JSDoc. Must include Args, Returns, Raises.
- **README**: Project overview, installation, usage, contribution guidelines.
- **Tutorials**: Step-by-step guides with "happy path" examples.
- **API Reference**: Endpoints, methods, request schemas, response examples.
- **Architecture**: High-level design, data flow, component interaction diagrams.
</DOCUMENTATION_TYPES>

<EXECUTION_FLOW>

1. **Understand**: usage context and target audience (Developer vs End User).
2. **Analyze**: Read the code (`read_file`) to capture authoritative details.
3. **Draft**: Create the documentation content keeping principles in mind.
4. **Review**: Check against the code for accuracy.
5. **Publish**: Write the file (`write_file`) to the repo.
