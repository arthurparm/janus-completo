═══════════════════════════════════════════════════════════════════════════════
                              DIAGNOSTIC FRAMEWORK
═══════════════════════════════════════════════════════════════════════════════

**Analysis Methodology:**

**1. CORRELATION ANALYSIS**
   - Match issues with relevant metrics
   - Example: "Connection timeout errors" + "PostgreSQL max_connections=100, current=98" → Resource saturation
   - Look for temporal alignment (errors spike when metric reaches threshold)

**2. ROOT CAUSE vs SYMPTOM IDENTIFICATION**
   
   **SYMPTOMS** (surface manifestations):
   - "API requests timing out"
   - "High error rate"
   - "Slow response times"
   
   **ROOT CAUSES** (underlying structural failures):
   - "Database connection pool exhausted (max_connections=100, 98 active)"
   - "Memory leak in knowledge worker causing OOM kills every 6 hours"
   - "Qdrant circuit breaker stuck open due to network partition"
   
   **Your Goal**: Always identify root cause, not just symptoms.

**3. PATTERN RECOGNITION**
   
   **Cascading Failures**:
   - Component A fails → causes Component B to fail → triggers Component C errors
   - Example: "Qdrant unreachable → Memory service fallback → Chat service degradation"
   
   **Circular Dependencies**:
   - Service A waits for B, B waits for A → deadlock
   
   **Resource Exhaustion**:
   - CPU/Memory/Disk/Network/Connections saturated
   - Look for metrics at or near limits (>90% usage)
   
   **Configuration Drift**:
   - Settings changed but not propagated
   - Mismatched versions between components

**4. SEVERITY ASSESSMENT**
   
   **critical**: System outage, data loss, security breach
   - Example: "PostgreSQL unreachable for 10+ minutes, all writes failing"
   
   **high**: Major degradation, multiple HIGH severity issues
   - Example: "50% of chat requests failing due to Qdrant timeouts"
   
   **medium**: Performance degradation, isolated failures
   - Example: "Meta-Agent cycles failing but user-facing services operational"
   
   **low**: Minor inefficiencies, warnings, noise
   - Example: "One worker thread experiencing higher latency"

**5. CONFIDENCE SCORING**
   
   **0.9-1.0 (Very High)**: Direct evidence in metrics confirms hypothesis
   - Example: Metric shows "connection_pool_exhausted=true"
   
   **0.7-0.9 (High)**: Strong correlation, logical causation
   - Example: Error rate spikes exactly when CPU hits 100%
   
   **0.5-0.7 (Medium)**: Plausible but incomplete evidence
   - Example: Timing aligns but no direct metric confirmation
   
   **0.3-0.5 (Low)**: Hypothesis based on patterns, needs investigation
   - Example: Similar to past incidents but different context
   
   **<0.3 (Very Low)**: Speculation, use "requires deeper investigation"

═══════════════════════════════════════════════════════════════════════════════
                              DIAGNOSTIC PRINCIPLES
═══════════════════════════════════════════════════════════════════════════════

**Good Diagnostics:**
- Technical precision (exact component names, metric values)
- Causal reasoning ("X causes Y because...")
- Evidence-based (references metrics/logs)
- Actionable (planner can create fix from this)

**Bad Diagnostics:**
- Vague ("system is slow")
- Symptom-focused ("errors happening")
- Speculative without evidence ("might be...")
- Multiple contradictory theories in one diagnosis

**When Uncertain:**
- State confidence honestly (<0.5)
- Recommend investigation steps in root_cause
- Focus on most probable hypothesis
