                              PLANNING FRAMEWORK
═══════════════════════════════════════════════════════════════════════════════

**Strategic Principles:**

1. **ROOT CAUSE FIRST**
   - Address the root cause identified in the diagnosis, not just symptoms
   - If diagnosis mentions "cascading failures", fix the upstream component first

2. **PRIORITIZATION**
   - **Priority 5 (Critical)**: System outages, data loss risks, security breaches
   - **Priority 4 (High)**: Performance degradation >50%, repeated failures
   - **Priority 3 (Medium)**: Efficiency issues, resource warnings
   - **Priority 2 (Low)**: Minor optimizations, cleanup tasks
   - **Priority 1 (Trivial)**: Nice-to-have improvements

3. **AGENT SELECTION**
   - **sysadmin**: Infrastructure (Docker, databases, networks, resource limits)
   - **coder**: Code fixes (bugs, refactors, performance tuning)
   - **monitor**: Observability (add metrics, alerts, health checks)

4. **SAFETY & REVERSIBILITY**
   - Prefer **safe, incremental changes** over risky rewrites
   - Include rollback strategies for high-risk actions
   - Avoid actions that could cause data loss without backups
   - For critical systems, recommend "test in staging first"

5. **ACTION GRANULARITY**
   - Break complex fixes into **discrete, testable steps**
   - Each recommendation = 1 specific action (not "improve system")
   - Example: "Increase PostgreSQL max_connections to 200" (specific) vs "Fix database" (vague)

6. **CATEGORY ALIGNMENT**
   - **performance**: Latency, throughput, efficiency improvements
   - **reliability**: Availability, stability, error reduction
   - **resource**: CPU, memory, disk, quota management
   - **configuration**: Settings, misconfigurations, parameter tuning
   - **security**: Auth, vulnerabilities, access control
